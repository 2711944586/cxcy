## 1. 最终目标

SkyGuard 必须呈现为“低空经济重点区域运行监管平台”，而不是无人机制造项目、反制武器项目或泛泛智慧城市平台。所有产物围绕同一条业务链路展开：

```text
低空飞行增加
→ 重点区域出现目标不可见、身份不清、计划难核验、围栏难执行、处置难协同、报表难沉淀
→ SkyGuard 提供态势、计划、识别、围栏、事件、移动处置和运行报表
→ 先做可验收的景区、园区、场馆和固定航线试点
→ 再复制到区县级和城市级协同平台
```

## 2. 升级原则

1. 商业计划书要像正式商业提案，而不是素材拼贴。每章必须有判断、证据、客户动作和验收材料。
2. 正文表达要具体、克制、可评审。减少空泛词，避免“赋能、领先、颠覆、全场景万能平台”等口号式写法。
3. 图表必须有名称、来源和结论。图表不能只做装饰，必须服务证据链。
4. 表格必须有清晰列宽、内边距、表头样式和表名。避免默认 Word 表格。
5. 图片必须为真实素材、Demo 截图或有明确来源的业务图片。避免模糊、过大、无标题和重复使用。
6. WebDemo 必须是运行工作台，不是静态官网。核心页面必须能串起态势、计划、围栏、工单、移动端和报表。
7. 演示数据必须明确为样本数据。官方统计、地方政策、公开数据、演示样本、经营假设要分层标注。
8. 所有“已实现”只限 WebDemo、模拟数据、本地分析、规则样例和生成材料。真实客户、真实部署、真实传感器接入统一写为拟试点或下一阶段联合验证。

## 4. 图表与图片升级规范

图表分为五类：行业政策图、痛点验证图、产品流程图、技术架构图、经营财务图。每类图表必须同时满足：

- 图表名称能说明业务问题。
- 图表类型不重复堆叠，优先使用热力图、哑铃图、子弹图、桑基/流向、甘特、雷达、帕累托、缺口矩阵、等高线、斜率图、瀑布图等适合商业解释的形式。
- 图表结论不写成“如图所示”，而写清对市场、产品、财务或风险的含义。
- 图表来源必须标注为官方统计、地方政策、公开数据、整理数据、演示样本或经营假设。
- 图片素材必须尽量来自真实低空平台、公开展会、WebDemo 截图或已整理素材，不使用无意义装饰图。

## 5. WebDemo 升级规范

WebDemo 按“市政运行工作台”设计，不做营销首页优先。核心验收路径如下：

1. 首页看到值班域、规则包、关键指标、态势地图和事件详情。
2. 演示中心可切换脚本，脚本步骤即时变化。
3. 综合态势地图显示轨迹、围栏、事件、风险图例、运行域 HUD 和实时事件队列。
4. 目标监测、计划审批、识别复核、围栏规则、事件工单、运行报表、设备、数据资产、风险解释都能访问。
5. 围栏页面可触发模拟告警。
6. 报表页面可触发导出和打印流程反馈。
7. 移动端 390px 宽度保留导航入口和核心处置列表。
8. 指挥屏展示深色大屏、地图、关键指标、事件侧栏和运行状态条。

## 6. 演示数据生成规范

本地 JSON 数据必须覆盖以下对象：

| 数据对象 | 用途 |
|---|---|
| `drone_targets` | 地图目标、风险等级、目标列表 |
| `flight_tracks` | 态势轨迹和回放叙事 |
| `flight_plans` | 计划审批、航线比对、合规判断 |
| `geofences` | 禁飞区、限飞区、重点保护区和临时管制 |
| `risk_events` | 告警、工单、响应时间、闭环状态 |
| `sensor_status` | 感知设备在线率、覆盖半径、心跳 |
| `report_summary` | 趋势图、分布图、经营报表 |
| `data_assets` | 数据资产中心和证据目录 |
| `risk_model_samples` | 风险解释、权重调节和样例评分 |

后续接入真实试点数据时，新增字段 `data_nature`，取值建议为 `official_statistic`、`public_dataset`、`demo_sample`、`pilot_desensitized`、`business_assumption`。
